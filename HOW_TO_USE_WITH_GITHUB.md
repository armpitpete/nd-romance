# How to Use This Repo with GitHub

## Purpose

This repo is the working Markdown source for the ND Romance pack.

Use GitHub for:

- version history
- issue tracking
- clean Markdown files
- samples and templates
- review notes

Use Google Drive for:

- original zip archive
- backup copies
- exports
- easy file storage outside GitHub

---

## Start path

Open these first:

1. `PROJECT_STATUS.md`
2. `FIRST_USE.md`
3. `templates/README.md`

Then copy blank forms from:

```text
templates/
```

Do not start by reading every file in `active_pack/`.

---

## Folder roles

## `active_pack/`

Guidance files and full control files live here.

Use these when you need explanation or deeper checks.

Do not treat these as the easiest working copies.

## `templates/`

Blank copyable working forms live here.

Use these for new romance projects.

Copy them before filling them in.

## `sample_projects/`

Worked examples live here.

Use these to see how the pack behaves on a real sample.

Do not mix sample files into `active_pack/`.

---

## Simple working method

For a new romance project:

1. Copy `templates/mini_design_blank.md`.
2. Fill it in roughly.
3. Copy `templates/structure_spine_blank.md`.
4. Fill only the simple structure path first.
5. Copy `templates/arc_map_blank.md`.
6. Map the relationship movement.
7. Copy `templates/scene_card_blank.md`.
8. Fill one scene card.
9. Draft one scene.
10. Review the scene against the scene card.

Do not open the checklist, triage sheet, or revision workflow until you have a draft or a clear problem.

---

## Issue workflow

Use GitHub issues for one small step at a time.

Good issue examples:

```text
Add filled scene card for Chapter 2
Review Chapter 1 draft v0.2
Create Beat Tracker example for Quiet Hours
Add changelog
```

Bad issue examples:

```text
Improve everything
Rewrite the whole pack
Make it better
Fix romance
```

Each issue should have:

- one task
- one output file or clear result
- a good-enough rule
- a close point

---

## Version rule for samples

Preserve earlier drafts.

Use filenames like:

```text
Quiet_Hours_Chapter_01_The_Broken_Lamp_Draft.md
Quiet_Hours_Chapter_01_The_Broken_Lamp_Draft_v0.2.md
Quiet_Hours_Chapter_01_The_Broken_Lamp_Draft_v0.3.md
```

Do not overwrite a sample draft unless the change is tiny and explicitly intended.

---

## What to edit

Edit or add files in:

```text
templates/
sample_projects/
PROJECT_STATUS.md
FIRST_USE.md
README.md
```

Edit `active_pack/` only when improving the pack itself.

---

## What not to do

Do not:

- store large zip exports in GitHub
- mix Google Drive archive files into the repo
- edit sample files as if they are templates
- edit templates as if they are filled projects
- add more theory before testing the existing workflow
- create huge issues with many unrelated tasks

---

## Current best next step

When unsure, do this:

```text
Open PROJECT_STATUS.md and do the current recommendation.
```

That keeps the repo moving one controlled step at a time.
